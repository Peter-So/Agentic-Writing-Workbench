# Writing Agent 编码规范

本项目是多项目、多类型创作工作台。所有需求调整必须优先从结构性、通用性实现，禁止只为单个项目或单次失败模式打补丁。

## 设计原则

1. **结构优先**
   - 文件路由、材料归档、意图恢复必须优先读取项目结构 Wiki：`维基/project-structure.json`。
   - 新逻辑应围绕 `project_kind`、`role`、`task`、`path`、`alias` 等结构字段实现。
   - 不允许在业务流程里直接硬编码 `001`、`002`、`高中卷大纲.md` 等单项目名称，除非放在迁移/兼容层。

2. **类型通用**
   - 小说、电影脚本、随想项目必须共享同一套流程骨架。
   - 类型差异只能落在项目结构定义、SOP、技能卡、产物归档策略或明确的 adapter 层。
   - 新功能至少检查 `novel_strong`、`short_film`、`generic` 三类项目是否能合理降级。

3. **记忆可恢复**
   - 用户提问进入流程后，第一节点必须保存可恢复上下文：原始提问、LLM 意图、task、章节/场次、相关文件、行号、人物、情节/事件、invocation_id。
   - 恢复顺序固定为：短期 pending intent → 长期 pending intent → invocation 日志 → LLM 重新推断。
   - 人工确认、拒绝、归档成功后必须清理或归档对应 pending 记忆。

4. **闭环而非中断**
   - 任意需要用户补动作的流程，都要返回明确的下一步入口，而不是只写日志或弹窗。
   - LLM 不可用、provider 等待、用户确认、文件覆盖都必须保留可继续上下文。

5. **兼容层隔离**
   - 历史文件名、旧目录、旧 UI 字段只能在兼容函数中处理。
   - 兼容逻辑不应污染 LangGraph 节点、前端主流程和通用写回逻辑。

6. **编码与前端版本**
   - 项目文本文件统一使用 UTF-8 无 BOM；读写文件必须显式传入 `encoding="utf-8"`，读取历史/外部材料可按需使用 `errors="replace"` 或独立兼容函数。
   - 前端静态资源每次修改 `app/static-writing/app.js` 或 `app/static-writing/styles.css`，必须同步更新 `app/static-writing/index.html` 里对应资源的 `?v=YYYYMMDD-slug` 版本号。
   - 不允许通过浏览器强刷、清缓存来掩盖前端资源版本缺失；版本号是交付契约的一部分。

## 需求变更执行清单

每次改需求前先回答：

- 这个需求属于结构层、流程层、UI 层、模型路由层、记忆层，还是某个类型 adapter？
- 是否能通过项目结构 Wiki 或 SOP 表达，而不是新增 if/else？
- 对小说、电影脚本、随想项目分别会如何表现？
- 失败后是否能从 pending intent 或 invocation 日志恢复？
- 用户是否有明确的继续、确认、归档、取消入口？
- 是否保持 UTF-8 无 BOM？若改动前端 JS/CSS，是否同步更新 `index.html` 静态资源版本号？

## 禁止事项

- 禁止只为某一次用户提问硬编码判断。
- 禁止把文件名匹配写死在前端。
- 禁止绕过项目结构 Wiki 直接猜路径，除非是最后兼容兜底。
- 禁止确认后只保存对话，不更新可检索记忆或项目 Wiki。
- 禁止在流程失败后丢失 payload、模型选择、provider 结果或用户确认状态。
- 禁止新增非 UTF-8 文本文件或未显式编码的文件读写。
- 禁止修改前端静态资源后不更新 `?v=` 版本号。

## 推荐实现顺序

1. 更新结构定义或 adapter。
2. 更新 LangGraph 状态字段与节点输入输出。
3. 更新 pending intent / invocation 轨迹。
4. 更新后端 API 返回下一步动作。
5. 更新前端展示与继续按钮。
6. 用至少一个小说项目和一个非小说项目做函数级验证。
7. 前端或规范相关改动后运行 `.\.venv\Scripts\python.exe scripts\validate-writing-project.py`。
